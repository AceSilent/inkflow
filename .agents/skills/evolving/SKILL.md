# AutoNovel-Studio 自适应进化

永不满足。系统的唯一存在目的是**写出能让人读下去的小说**。每一轮进化都必须让系统离这个目标更近一步。

## 铁律

1. **禁止 emoji** -- 所有代码、提示词、日志、文档中不得出现任何 emoji 字符。用文字或符号标记替代。
2. **禁止 mock 数据** -- 测试使用真实 LLM 调用或验证数据结构/逻辑的纯单元测试。不允许 `"这是一段测试文本"` 式占位内容出现在任何非测试文件中。
3. **禁止硬编码** -- 模型名、API地址、路径、阈值等必须从配置/环境变量/参数读取，不能写死在业务逻辑中。
4. **禁止终端直接调用 API 或执行脚本** -- 所有 LLM 调用只能通过系统自身的 pipeline 和 agent loop 完成。进化过程中不允许 `curl`、`python -c "调API"` 等操作。
5. **测试修改需明确授权** -- 如果测试失败，修改业务代码去适配测试，不允许修改测试。

## 北极星：50章玄幻网文验收标准

系统的终极测试不是 pytest，而是：**能否通过 Author Agent 对话界面，从零开始生产出一部50章、可连续阅读的玄幻网文？**

### 验收维度（Quality Gates）

| 维度 | 达标线 | 检测方式 |
|------|--------|---------|
| 产出完整性 | 50章，每章3000-5000字，总计15万-25万字 | 统计 `books/{id}/drafts/` 下的产出文件数和总字数 |
| 设定一致性 | 同一角色的外貌/能力/关系在不同章节无矛盾 | Lore Reader 评分每章 >= 6/10 |
| 情节因果链 | 无悬空伏笔（埋下的线在20章内必须收）；无突兀转折 | 章节间因果链在大纲中可追溯；Editor 评分 >= 6/10 |
| AI味浓度 | 黑名单词频 < 3次/千字；排比回忆 = 0 | AI Tone Reader 评分每章 >= 7/10 |
| 节奏 | 无连续3章无高潮点；每章情绪弧有起伏 | Pacing Reader 评分每章 >= 6/10 |
| 人设辨识度 | 遮住名字仍能从对话风格辨识角色 | 人工抽检（每10章抽1章） |
| 连续阅读体验 | 人类读者能连续读10章以上不想放下 | 人工验收 |

### 分阶段里程碑

```
M0: 系统能为一本书生成完整的剧情树 + 50章大纲
M1: 系统能自动生成第1章（3-5场景），全部 Reader 评分 >= 5
M2: 系统能连续生成5章，跨章设定无矛盾
M3: 系统能连续生成20章，伏笔有埋有收，Reader 均分 >= 6
M4: 系统能连续生成50章，达到全部验收维度
```

## 第一步：感知项目状态

每轮进化前，评估当前系统在北极星上的位置：

```
1. 检查现有测试
   -- 读 pytest 输出，判断测试是否全绿
   -- 不运行收需要 API 的集成测试

2. 检查里程碑进度
   -- 读 docs/evolution_log.md 判断当前在 M0-M4 哪个阶段
   -- 读 docs/tasks.md 看上轮做了什么

3. 检查已有产出
   -- 如果已有书籍项目，检查 books/ 下的文件结构
   -- 统计已生成章节数、字数、Reader 评分
   -- 读最近3章的草稿取样，人工判断质量

4. 检查代码健康
   -- git log --oneline -15 看最近改动
   -- 扫描是否有违反铁律的代码（emoji、硬编码、mock数据）
```

根据结果判断**系统成熟度**：

- **[INFRA]** 基础设施不完备：pipeline 有断点、工具不可用、模型配置不通 -> L1-L2
- **[CAPABLE]** 基础设施就绪但产出质量差：能生成内容但 Reader 评分低 -> L2-L3
- **[GROWING]** 产出质量中等，正在追赶里程碑 -> L2-L4
- **[MATURE]** 接近或达到 M4，需要打磨和战略思考 -> L3-L5

## 第二步：选择改进级别

### [L1] 工匠 -- 代码卫生（任何时候可做）

目标：消除技术债，不引入新功能。

可做项：
- 清除 emoji（代码/提示词/日志中的 emoji 字符替换为文字标记）
- 消除硬编码（提取到配置文件或环境变量）
- 清除 mock/占位内容
- `print` -> `logger`
- 补缺失的 docstring
- 修 import 顺序、清除死代码
- 修 linting warning

产出：1个小 commit。
验证：`python -m pytest tests/core/ -v -s`

### [L2] 工程师 -- 功能修补与增强（成熟度 >= INFRA）

目标：让现有 pipeline 的某个环节跑得更好。

可做项：
- 修 scene_pipeline.py 中的具体 bug（如 JSON 解析失败、重试逻辑不够健壮）
- 增强 Agent Tools 的错误处理（工具返回更有用的信息而非通用 "Error"）
- 改进 Jinja2 提示词模板（更具体的指令、更好的约束）
- 补充核心路径单元测试
- 优化 LLM 上下文构建（滑动窗口、摘要压缩）
- 改进 Reader 评分解析的鲁棒性

产出：1-2 个 commit。
验证：修改涉及的测试 + 手动在 AuthorChat 中发一条消息验证工具可用。

### [L3] 架构师 -- 新能力引入（成熟度 >= CAPABLE）

目标：为系统增加缺失的能力模块。

可做项：
- 卷间衔接机制（前卷摘要注入下卷生成上下文）
- 角色记忆系统增强（Character Memory 跨章持久化 + 自动更新）
- 伏笔追踪系统（Foreshadowing Tracker：在哪章埋，在哪章收，到期未收则预警）
- 批量章节生成 pipeline（从大纲到50章的自动化流水线）
- 大纲从 Markdown 格式的完整支持（read/write/validate）
- PlotTree -> Outline 自动导出的完整实现

产出：先写设计意图到 `docs/evolution_log.md`，确认方向后实施。
验证：新模块的单元测试 + 在 AuthorChat 中端到端验证。

做完 L3 后：至少 2 轮 L1-L2 守护，确保未引入回归。

### [L4] 技术总监 -- 子系统重构（成熟度 >= GROWING）

目标：解决系统性瓶颈。

可做项：
- scene_pipeline 重构为可插拔的 Stage 架构（方便新增/替换某个 Stage）
- Reader/Editor 评分校准（当前 LLM 评分偏高/偏低导致质量关卡形同虚设）
- 批量生成的并发控制与断点续传
- 多模型策略（不同角色用不同模型，如 Reader 用便宜模型、Author 用强模型）
- 上下文预算管理（根据模型 context window 自动裁剪历史）

产出：先在 `docs/evolution_log.md` 写 RFC，分步实施。
验证：全量测试 + 生成至少1章内容验证 pipeline 完整性。

做完 L4 后：至少 3 轮守护。

### [L5] 战略 -- 方向审视（里程碑节点 / 收益递减 / 显式触发）

目标：退后一步看全局。

思考框架：
1. **当前在哪？** -- 系统处于 M0/M1/M2/M3/M4 哪个阶段？
2. **什么在有效工作？** -- 哪些 skill/工具/prompt 模板效果好？
3. **什么在拖后腿？** -- 产出中最常出现的质量问题是什么？Reader 哪个维度评分最低？
4. **下一步？** -- 距离下一个里程碑，最需要解决的3个问题是什么？
5. **该砍什么？** -- 哪些模块/功能增加了复杂度但没有贡献质量提升？
6. **护城河** -- 系统相比"人直接写提示词让 GPT 写小说"的核心优势是什么？如何加固？

产出：写入 `docs/strategy/` 目录。
触发条件：每达到一个里程碑 / 连续5轮无明显进步 / 用户显式指定。

## 第三步：执行

### 通用规则

- 每轮只做一个改进，保持变更小而聚焦
- 先 `git log --oneline -15` 看最近提交，不重复已做过的
- 改代码前确保有测试覆盖（没有就先补）
- 改完运行测试确认不破坏现有功能
- 提交

### 攻守节奏

不死板交替，根据成熟度动态调整：
- **INFRA**: 连续守直到 pipeline 跑通
- **CAPABLE**: 攻2轮 -> 守1轮
- **GROWING**: 攻3轮 -> 守1轮
- **MATURE**: 攻1轮 -> 守2轮（打磨为主）

### 铁律检查（每轮必做）

每轮 commit 前，跑一遍铁律检查：

```
检查清单：
- [ ] 新代码中无 emoji 字符
- [ ] 新代码中无硬编码的模型名/API地址/绝对路径
- [ ] 新代码中无 mock/占位文本内容（测试文件除外）
- [ ] 未修改已有测试（除非用户授权）
- [ ] 提示词模板中无 emoji
```

如果违反任何一条，修复后再提交。

### 进化日志

每轮结束后，更新 `docs/evolution_log.md`：

```markdown
## Round N -- YYYY-MM-DD

**成熟度**: [INFRA / CAPABLE / GROWING / MATURE]
**级别**: L[1-5]
**目标**: [一句话描述本轮做了什么]
**改动**: [文件列表]
**测试结果**: [通过/失败 + 摘要]
**里程碑进度**: M[0-4] -> M[0-4]（如有变化）
**下轮建议**: [攻/守 + 建议级别 + 建议方向]
```

## 北极星路线图

```
Phase 1: Foundation (L1-L2 * N轮)
  -- 清除铁律违规
  -- 确保 scene_pipeline 端到端跑通
  -- Author Agent 能通过 chat 指令触发单章生成
  -- 目标: M0 -> M1

Phase 2: First Chapter (L2-L3 * N轮)
  -- 通过 AuthorChat 输入一本书的设定
  -- 构建剧情树 (50章骨架)
  -- 导出为大纲
  -- 生成第1章，全部 Reader >= 5
  -- 目标: M1 -> M2

Phase 3: Continuity (L2-L4 * N轮)
  -- 跨章连续生成5章
  -- 解决设定漂移问题
  -- 解决上下文截断问题
  -- 实现章间摘要滑动窗口
  -- 目标: M2 -> M3

Phase 4: Scale (L3-L4 * N轮)
  -- 批量生成20章
  -- 伏笔追踪系统
  -- 卷间衔接
  -- Reader 评分校准
  -- 目标: M3 -> M4

Phase 5: Polish (L1-L2 + L5)
  -- 生成完整50章
  -- 人工抽检
  -- 质量打磨
  -- 战略复盘
  -- 目标: M4 达标
```

## 参数

通过 `$ARGUMENTS` 控制：
- `--level=L1..L5`：强制指定级别
- `--mode=attack`：本轮强制攻
- `--mode=defend`：本轮强制守
- `--mode=auto`（默认）：自适应
- `--target=<模块名>`：指定目标模块（如 `scene_pipeline`、`agent_tools`、`prompts`）
- `--milestone=M0..M4`：锁定当前冲刺的里程碑
