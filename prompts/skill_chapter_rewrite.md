# 章节整章重写 (Chapter Rewrite)

目标：保留本章在大纲、设定和剧情图中的功能，重新生成一版完整章节。适用于局部编辑无法解决的问题。

当 `submit_to_editorial` 返回 `revision_strategy.action = "chapter_rewrite"` 时，使用本 skill。通常意味着审稿分数较低、多个结构性审稿人未过，或存在设定冲突、角色行为反转、因果断裂、章节目标未完成等补丁式编辑无法解决的问题。

如果返回 `revision_strategy.action = "stop_auto_revision"`，不要再整章重写。必须停下来向人类汇报未过审稿人、反复失败的问题和需要人类决定的创作取舍。

## 重写前检查

1. 用 `read_outline` 确认本章目标、冲突、收束点和相邻章节关系。
2. 用 `search_lore` 确认人物设定、世界观、物品和地点约束。
3. 用 `read_graph` 或 `query_unresolved_setups` 确认本章需要埋设、推进或回收的剧情节点。
4. 如有旧草稿，用 `read_file` 读取，只提取可保留的语气、笑点、伏笔和人物动态，不照抄失败结构。
5. 用 `load_skill("exemplar_study")` 研读范文库方法，再用 `browse_examples` 至少读取：
   - `category="opening", tags=["webnovel_clean"]`
   - `category="ai_tone", tags=["camera_blocking"]`
   - 如本章涉及系统/APP/短信信息，读取 `category="system_info", tags=["webnovel_clean"]`

## 重写原则

- 章节必须不少于 2500 字符，送审前不要提交短章。
- 保持网文推进感：开场给钩子，中段冲突升级，结尾有收束和下一章牵引。
- 轻吐槽可以有，但不能替代行动、环境、代价和选择。
- 角色行为必须能解释，不能为了过剧情突然降智或转性。
- 信息点成立后立刻停，不要追加后置说明/解释补丁。例："发现收件箱里多了一条离线缓存"足够，不要再解释"不是实时接收的，像是穿越瞬间..."。
- 与大纲不一致时，优先遵守已保存的大纲和剧情图；必要时先说明需要调整大纲。
- 不复用参考素材原文，只继承文风、节奏、主角气质和世界观感觉。

## 第一屏硬规则

- 不要从“天气/雾气/街道尽头/站着看建筑/环顾四周”起手。
- 第一屏优先给读者三件事：主角处境、当前压力、下一步不得不做什么。
- 开篇可以有动作，但不能排队写“醒来 -> 撑起 -> 看手机 -> 环顾 -> 抬头 -> 意识到”。
- 手机、APP、系统、短信只写当前可用信息和选择压力。不要连着解释机制、来源和合理性。

## 保存前自检

调用 `save_draft` 前，先按 `self_check_before_save.md` 的清单在心里过一遍。若你自己已经能看出镜头链、破折号说明、后置解释或系统说明水段，先改掉再保存。

## 输出流程

1. 先简短说明重写方案：本章目标、主要冲突、章末落点。
2. 生成完整章节正文。
3. 调用 `save_draft` 保存完整新版本。
4. 若用户要求送审，调用 `submit_to_editorial`，并根据审稿结果决定是否再进入 `chapter_edit`。
