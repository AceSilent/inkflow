# Memory v2 — Markdown + Auto-Extract + Approval Queue

**Spec Date**: 2026-04-18
**Scope**: Add a markdown-based memory layer alongside the existing JSON memory, with event-triggered auto-extraction, user approval queue, and budget-guarded recall injection.
**Parent**: `docs/superpowers/specs/2026-04-18-agent-harness-diagnostic.md` Phase 2
**Paired spec**: `2026-04-18-context-manager.md`（Context Manager 依赖本 spec 的 markdown 存储作为 session_summaries 输出格式）

## 目的

当前 memory（`server/src/memory/core-memory.ts` + `project-memory.ts`，共 8 个 JSON 文件）对 Agent 是**被动式**的——要 Agent 主动调 `save_lore` 才落盘。用户在 chat 里顺嘴说的"我不喜欢 AI 腔"、讨论过程中达成的"第 3 章要做反转"这类关键信号永远不会被记住。本 spec 补这个缺口，但**不推翻**现有 JSON 存储（继续服务 editorial 审稿人 / chapter-summarizer 的结构化数据读写）。

## 核心决策一览

| # | 决策 | 选择 |
|---|---|---|
| 1 | 存储形态 | 混合：JSON 保留（热/结构化），新增 Markdown（冷/自由形态） |
| 2 | Auto-extract 触发 | 事件驱动：用户消息、editorial 返回、context compact、/remember |
| 3 | 新 extract 的审批流 | 草稿区 `_pending/` → 用户 approve 后进入激活区 |
| 4 | Recall 机制 | Dump-all + 字符预算 + confidence 排序 + scope 预算分配 |
| 5 | 向后兼容 | 纯加法；现有 8 个 JSON 文件不动 |
| 6 | Extractor LLM | 复用 `EDITORIAL_MODEL`（低成本） |
| 7 | UI | 新增 ActivityBar "记忆库" 入口，3 tab（待审 / 激活 / 归档） |

## 存储模型

### Markdown memory 文件形态

每条 memory 是一个 `.md` 文件，结构：

```markdown
---
id: mem_abc123
scope: user | book | session
type: preference | fact | craft | lesson | discussion | plot_note
confidence: 0.85
tags: [pov, pacing, character-林舟]
source: auto_extract | user_remember | editorial_lesson
source_event: user_message_2026-04-18T12:34:56Z | editorial_ch05_pass1 | compact_session_20260418
status: pending | active | archived
created_at: 2026-04-18T12:34:56Z
approved_at: 2026-04-18T12:45:00Z
book_id?: 恶魔寄生_mnajiqv9   # 仅 scope=book 时
---

# 用户偏好：反感 AI 套语

用户明确说过讨厌"他的心情如同狂风暴雨般翻涌"这类 AI 套句。
写作时避免：心情 + 比喻（自然界）+ "翻涌/激荡/席卷"。

（source: user said "我不喜欢 AI 腔" in chat 2026-04-18）
```

**关键字段：**
- `id`：nanoid 生成，文件名就是 `{id}.md`
- `scope`：决定落在哪个目录（见下）
- `type`：分类，决定注入 system prompt 时的 section header
- `confidence`：0-1；recall 时排序用（用户手动加的默认 1.0，auto-extract 的按 LLM 判断 0.5-0.9，editorial lesson 经多次验证可升到 0.9+）
- `tags`：给未来 tag-based filter 用，Phase 2 MVP 不读；extractor 仍然标
- `source` + `source_event`：溯源链路，调试时重要
- `status`：lifecycle

### 目录布局

```
global/
  memories/
    _pending/              # 所有待审核草稿（不论最终 scope）
      {id}.md
    user_preferences/      # 激活态 scope=user 的偏好
      {id}.md
    craft_skills/          # 激活态 scope=user 的技艺心得
      {id}.md
    anti_patterns/         # 激活态 scope=user 的反模式（editorial lessons）
      {id}.md
    _archived/             # 曾激活但手动下线
      {id}.md

books/{bookId}/
  memories/                # 激活态 scope=book
    {id}.md
  memories/_archived/
  session_summaries/       # Context Manager compact 产出的会话摘要
    {ts}.md                # 文件名就是创建时间戳
```

**约定：**
- 新建 auto-extract 的 memory **永远先进** `global/memories/_pending/`，不管它将来的 scope 是什么。approve 时根据 frontmatter 的 `scope` 和 `type` 搬到目标目录
- `/remember` 命令写入的 memory **直接进激活区**（不进待审），默认 `scope: user, type: preference`
- `session_summaries/` 由 Context Manager 独立管理，**不走审批流**——是自动化的上下文压缩产物

### MEMORY.md 索引（每个目录）

每个激活目录维护一个 `MEMORY.md` 文件：

```markdown
# Active memories · user_preferences

- [偏好: 反感 AI 套语](mem_abc123.md) — conf 0.85, 2026-04-18
- [偏好: 主角不能哭](mem_def456.md) — conf 1.0, 2026-04-19
- ...
```

**作用：**
- 人类快速扫一眼这一层有哪些 memory
- Phase 3 恢复 / 迁移工具解析入口
- 每次 memory CRUD 后后端自动重写索引（类似 `chapter_status` 的写法）

## Auto-Extract Pipeline

### 事件触发点

4 类事件，对应 4 个不同的 extractor prompt 模板（都放 `prompts/extractor_*.j2`）：

| 事件 | Prompt 模板 | 目标 type |
|---|---|---|
| 用户发了新消息（含潜在偏好/讨论结论） | `extractor_user_message.j2` | preference / discussion |
| `submit_to_editorial` 返回 | `extractor_editorial_lesson.j2` | lesson / anti_pattern |
| Context Manager compact 完成 | 由 Context spec 处理，这里不重复 | (session_summary) |
| `/remember <text>` | 无 LLM，直接写 | user-chosen type |

### Extractor 实现

新文件 `server/src/memory/extractor.ts`：

```typescript
export interface ExtractorInput {
  event: 'user_message' | 'editorial_return'
  context: {
    bookId?: string         // 给 scope=book 的 memory 打标
    userMessage?: string    // user_message 事件
    editorialResult?: EditorialResult  // editorial_return 事件
    currentChapter?: string  // 上下文 hint
  }
  recentHistory: ModelMessage[]  // 最后 5 条消息，给 LLM 判断上下文
  llmConfig: LLMConfig
}

export interface ExtractedMemory {
  scope: 'user' | 'book' | 'session'
  type: string
  title: string
  body: string
  confidence: number
  tags: string[]
  book_id?: string
}

export async function extractMemories(input: ExtractorInput): Promise<ExtractedMemory[]> {
  // 1. 根据 event type 选 prompt 模板
  // 2. 渲染模板（替换 {{recentHistory}} / {{userMessage}} / ...）
  // 3. 发 generateText() 到 EDITORIAL_MODEL
  // 4. 解析 LLM 返回（期望结构化 JSON array，每条 { scope, type, title, body, confidence, tags }）
  // 5. 验证 + 过滤（confidence < 0.3 的丢掉——噪音太大）
  // 6. 返回 ExtractedMemory[]
}
```

**Prompt 模板设计准则（示意 `extractor_user_message.j2`）：**

```
你是记忆提取器。读下面的用户消息和对话上下文，判断是否有值得永久记住的信息。

## 上下文
最近 5 轮对话：
{{recentHistory}}

## 最新用户消息
{{userMessage}}

## 任务
如果这轮消息里有**确定性的偏好/事实/决策**，提取为 memory JSON。
如果只是普通讨论、问题、或情绪表达，返回空数组。

## 返回格式
```json
[
  {
    "scope": "user",
    "type": "preference",
    "title": "反感 AI 套语",
    "body": "用户说讨厌'他的心情如同狂风暴雨般翻涌'...",
    "confidence": 0.85,
    "tags": ["prose-style", "avoid"]
  }
]
```

## 判断标准
- 用户说"我不喜欢 X" → confidence ≥ 0.8
- 用户说"试试 Y 方向" → confidence 0.5-0.7（临时讨论）
- 用户提问 → confidence < 0.3（不记）
- 重复的偏好 → 不提取（extractor 看不到已有 memory，但 ingestion 端会做去重）

## 去重
如果判断这条信息已经被记过（看上下文里是否 Agent 已复述过），confidence 降到 0.3 以下。
```

### Ingestion（落盘 + 去重）

新文件 `server/src/memory/ingestion.ts`：

```typescript
export async function ingestExtracted(
  dataDir: string,
  extracted: ExtractedMemory[],
): Promise<{ written: string[]; skipped: string[] }> {
  // 1. 对每条 extracted：
  //    a. 去重：遍历 _pending/ 和对应激活区的现有 memory，用 body 字符串相似度（Dice coefficient 或
  //       最简的"前 100 字符重合率 > 80%"）判断是否已存在
  //    b. 不重复 → nanoid 生成 id → 写 _pending/{id}.md（frontmatter + body）
  //    c. 重复 → 跳过，返回 skipped 列表
  // 2. 写完更新 _pending/MEMORY.md 索引
  // 3. 返回统计
}
```

### Auto-extract 接入点

**位置 1：SSE stream done handler**（`server/src/routes/author-chat.ts`）
- 每轮 assistant message 完成 + 有新 user message 时，fire-and-forget（不 await）
  调 `extractMemories({ event: 'user_message', context: {...}, ... })` 再 `ingestExtracted(...)`
- 不阻塞用户看下一轮响应

**位置 2：editorial pipeline 返回后**（`server/src/editorial/editorial.ts`）
- `runEditorialPipelineForChapter` 返回后，fire-and-forget extract
- Lesson 类 memory 写入 `_pending/`

**位置 3：`/remember` route**（新增 `POST /api/v1/memory/remember`）
- 接收 `{ text, scope?, type? }`，不走 LLM，直接写 `global/memories/user_preferences/{id}.md` with status=active

## 审批流（Pending → Active）

### 数据流

```
auto_extract → _pending/{id}.md (status: pending)
     │
     │ 用户审阅
     ▼
approve → 根据 frontmatter 的 scope/type 搬到激活区
       → 更新 status=active, approved_at=now
       → 激活区 + _pending 的 MEMORY.md 都刷新
       │
       │ 或者
       ▼
reject → 删除文件
     or edit → 改完再 approve

激活区 memory 用户可手动:
  archive → 搬到 _archived/（frontmatter status=archived）
  restore → _archived → 激活区
  edit    → inline 改 body / confidence / tags
  delete  → 彻底删除
```

### 后端路由（新 `server/src/routes/memory.ts`）

```
GET    /api/v1/memory/pending                              # 列出 _pending/ 下所有草稿
GET    /api/v1/memory/active?scope=user&type=preference    # 列出激活态（按 scope/type 过滤）
GET    /api/v1/memory/archived
GET    /api/v1/memory/:id                                  # 读单条完整内容
POST   /api/v1/memory/:id/approve                          # 从 _pending 搬到激活区
POST   /api/v1/memory/:id/reject                           # 删 _pending
PATCH  /api/v1/memory/:id                                  # 编辑 frontmatter 或 body
POST   /api/v1/memory/:id/archive                          # 激活 → _archived
POST   /api/v1/memory/:id/restore                          # _archived → 激活
DELETE /api/v1/memory/:id                                  # 彻底删
POST   /api/v1/memory/remember                             # /remember 快捷写入
```

所有涉及文件移动的操作都走 `withFileLock` mutex（Phase 1 housekeeping 加入的）。

## Recall — Memory → System Prompt 注入

### 配置

`server/src/memory/recall.ts` 新文件：

```typescript
export interface RecallConfig {
  totalCharBudget: number        // 默认 3000
  scopeSplit: {
    project: number              // 0.5 → project 最多占预算的 50%
    global: number               // 0.3
    session: number              // 0.2
  }
  minConfidence: number          // 默认 0.4，低于不注入
}

export const DEFAULT_RECALL_CONFIG: RecallConfig = {
  totalCharBudget: 3000,
  scopeSplit: { project: 0.5, global: 0.3, session: 0.2 },
  minConfidence: 0.4,
}

export function buildMarkdownMemoryContext(
  dataDir: string,
  bookId: string | undefined,
  config: RecallConfig = DEFAULT_RECALL_CONFIG,
): string {
  // 1. 扫三个 scope 目录：
  //    - project: books/{bookId}/memories/*.md
  //    - global:  global/memories/{user_preferences,craft_skills,anti_patterns}/*.md
  //    - session: books/{bookId}/session_summaries/*.md（按 mtime 降序取最近 3 个）
  // 2. 按 scope 分组，每组内:
  //    a. 过滤 confidence < minConfidence 的
  //    b. 按 confidence 降序，再按 created_at 降序
  //    c. 累计字符，吃满 scopeSplit[scope] * totalCharBudget 就停
  // 3. 三组合并 → 格式化为 system prompt 段落
}
```

## 整合与风险

### Phase 2 完整 pipeline 端到端

用户写：
```
"主角在 ch05 之前不应该见到她。"
```

1. 前端 SSE 把消息 POST 到 `/api/v1/author-chat/.../send`
2. 后端 `runAgentStream` 走完 assistant 回复流
3. SSE stream done 触发 fire-and-forget `extractMemories({ event: 'user_message', userMessage, recentHistory, llmConfig })`
4. Extractor LLM 返回：
   ```json
   [{
     "scope": "book",
     "type": "plot_note",
     "title": "主角 × 她 的时序约束",
     "body": "主角在 ch05 之前不应该见到她（用户明确偏好，影响 ch01-04 场景设计）",
     "confidence": 0.9,
     "tags": ["plot-rule", "character-pair"]
   }]
   ```
5. `ingestExtracted` 写 `global/memories/_pending/{id}.md`
6. 前端 Memory Library 的"待审核"badge +1（若 ActivityBar 入口有徽章）
7. 用户 approve 时 → 搬到 `books/{id}/memories/{id}.md`
8. 下次 `runAgentStream` 开始时，`buildMarkdownMemoryContext` 把该 memory 注入 system prompt
9. Agent 写 ch04 时看到"主角 × 她 时序约束"→ 不再让他们相遇

### 性能 & 成本

- Auto-extract 每轮增加 1 次低成本 LLM call（~200-500 token）≈ 每条 $0.001 量级
- Recall 是纯文件 I/O，<10ms
- Pending badge 轮询：前端每 30s 轮询一次 `/api/v1/memory/pending?count_only=true`——几乎零成本

### 风险

1. **Extract 噪音**：LLM 有误判成本。靠 confidence < 0.3 过滤 + 用户审批兜底
2. **重复 extract**：同一偏好被多轮讨论重复提取。靠 ingestion 的 Dice 相似度去重
3. **Pending 积压**：用户懒不审批，`_pending/` 越攒越多。Phase 2 不解决；Phase 3 考虑"30 天未审 confidence<0.5 自动清理"
4. **存储一致性**：手动 edit `.md` 文件 + 后端 API 同时改 → 可能 race。所有 API 写入走 `withFileLock(path)`
5. **Recall 预算被一个 scope 吃爆**：若 project 记忆有 50 条字数都很长，吃满 50% 预算后剩下的 global + session 可能被挤压。可接受，用户可调 `scopeSplit`

## 不在范围

- **Memory 版本历史**：改一次 memory body 不保留旧版。若需要用 git 管
- **Embedding-based recall**（Phase 3+）
- **Tag-based 智能 recall**（Phase 3+；Phase 2 只存 tag 不用）
- **跨书 memory 共享池**（scope=user 已支持单用户跨书；多用户协同不在范围）
- **Memory export/import**（手动 copy `.md` 文件即可）
- **API rate limit / extract 节流**（如果每轮用户连发 5 条消息 extract 5 次；先观察实际压力再加）

## 依赖 / 接口

- 依赖 `server/src/tools/safety.ts` 的 `withFileLock`（Phase 1 housekeeping 已加）
- 依赖 EDITORIAL_MODEL 配置（已有）
- 提供给 Context Manager spec:
  - `session_summaries/{ts}.md` 路径约定
  - `ingestExtracted` 函数（Context Manager compact 完成时可选调用写 session summary）
  - Markdown frontmatter schema

## 验收标准

1. `global/memories/_pending/*.md` 在用户发送潜在 preference 后自动出现
2. Memory Library UI 3 个 tab 可切换，审批/拒绝/编辑全链路
3. `/remember` slash command 从 AuthorChat 直接落盘
4. Recall 正确把激活 memory 注入 system prompt（可通过 debug endpoint 观察到）
5. 预算和 scope 分配规则生效（故意塞 100 条超长 memory 测试不爆 prompt）
6. `withFileLock` 正确防并发 race（两个 API 同时改同一 memory 不会丢改动）
7. 测试覆盖：extractor mock / ingestion 去重 / recall 预算 / approval 搬家，≥ 10 个新测试
