# AutoNovel Studio - Agentic Workflow Architecture Redesign
**Date**: 2026-03-25

## 1. Core Paradigm Shift
Transitioning the AutoNovel Studio from a "prompt-based chat assistant" to a **State-Driven Agentic Workflow**. The system will operate as an autonomous pipeline where AI Agents independently execute tasks, use tools to gather context, and collaborate, with the human acting as the final approver and director.

## 2. Three Iron Rules of Development
The following rules must be strictly adhered to during the implementation of this redesign:
1.  **No Mock Data allowed**: All data shown in the UI or processed by the system must be the actual, real data generated by the backend components. The final deliverables must run strictly on real data.
2.  **Timely Process Management**: All completed bash processes must be explicitly stopped/killed in a timely manner to prevent zombie processes from clogging up the terminal and blocking new command executions.
3.  **Use Background Bash Tasks**: Always run bash commands in the background to prevent blocking or freezing the main working thread and keeping the system responsive.

## 3. Key Architecture Components

### 2.1 Agent Roles & Responsibilities
*   **Author Agent (作者)**: The core creator. Upgraded from a simple text generator to a fully autonomous agent.
    *   **Capabilities**: Can autonomously use tools (Skills) to read outlines, query character lore, read previous chapters, and study "Excellent Examples".
    *   **Context Strategy**: "Golden Ratio" — The system injects absolutely critical context (current scene prompt), but the Author explicitly calls tools to retrieve specific details (e.g., `check_character_status("Vera")`) before drafting.
*   **Editorial Department (编辑部)**: Formerly "Readers". Acts as the internal quality control gate.
    *   **Capabilities**: Reviews the Author's drafts against the outline and lore. If quality is poor or out-of-character (OOC), the Editor generates a critique report and rejects the draft back to the Author.
*   **Lore Agent (考据档案员)**: A new, passive background agent.
    *   **Capabilities**: After a chapter is finalized by the human, this agent silently reads the chapter and updates `characters.json`, `world_setting.json`, and internal wikis to maintain a globally accurate memory state as the word count grows.

### 2.2 Task State Machine (The Backbone)
To support long-running processes (e.g., an Author rewriting a chapter 3 times based on Editorial feedback), execution state must be persisted.

*   **Storage**: Local JSON Task Logs (`books/<book_id>/tasks/<task_id>.json`).
*   **States**:
    1.  `DRAFTING`: Author is researching and writing.
    2.  `EDITORIAL_REVIEW`: Draft sent to Editors. (Can cycle back to `DRAFTING` up to 3 times before failing).
    3.  `HUMAN_APPROVAL_PENDING`: Editor-approved draft is waiting for the user.
    4.  `COMPLETED`: Human approved. Chapter published. Lore Agent triggered.
    5.  `REJECTED`: Human rejected with feedback (Cycles back to `DRAFTING` with feedback).

### 2.3 Context & Data Isolation
*   Strict physical folder isolation per book (`books/<book_id>/`).
*   **Dual-layer Memory Retained**: The `chat_context.json` memory compression layer is kept for discussion history, but drafting context heavily relies on explicit Tool calls by the Author retrieving data from `lore/` and `outlines/`.
*   **Iceberg Engine**: Maintained as a callable Skill (`search_iceberg_notes()`) for the Author to uncover deep subtext.

## 4. Workflow Example: Writing a Chapter
1.  **Human Command**: User creates a Task: "Write Chapter 5 based on Scene Node 3."
2.  **Author Preparation**: Author Agent starts. It calls `read_outline(volume=1)` and `read_lore()`.
3.  **Author Drafting**: Author writes Draft v1.
4.  **Editor QC**: Draft v1 goes to Editor. Editor flags pacing issues.
5.  **Author Revision**: Author receives Editor critique, rewrites Draft v2.
6.  **Editor Pass**: Editor approves Draft v2. Task state shifts to `HUMAN_APPROVAL_PENDING`.
7.  **Human Review**: User reads Draft v2 in the UI. User says "Add more dialogue at the end."
8.  **Final Polish**: Author processes human feedback -> Draft v3 -> Editor -> Human Approves.
9.  **Publishing**: Draft v3 becomes official Chapter 5 text.
10. **Lore Update**: Background Lore Agent scans Chapter 5, discovers a new magical item, and appends it to `world_setting.json`.

## 5. Phased Implementation Plan
*   **Phase 1: Task State Machine Infrastructure** — Build the JSON task log system and the basic run-loop.
*   **Phase 2: Author Agent Refactoring & Skills** — Transition the Author to a tool-using agent capable of reading local files.
*   **Phase 3: Editorial Rejection Loop** — Implement the inner review loop and the Max Retries failsafe.
*   **Phase 4: Lore Agent Pipeline** — Implement the background task for knowledge extraction.
*   **Phase 5: UI/Frontend Overhaul** — Update the React frontend to display the async Task Queue, human approval gates, and draft reviews.
*   *(Future Phase: Feishu API integration for remote approval of `HUMAN_APPROVAL_PENDING` tasks).*
