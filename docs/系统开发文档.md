# AutoNovel-Studio 系统开发文档 (v1.0)

## 1. 架构总览 (Architecture Overview)

本系统摒弃LangChain等黑盒框架，采用**纯Python + 状态机（State Machine） + 全文档工作流（File System as Database）**架构。核心逻辑类似于生成对抗网络（GAN）：由“作者Agent”生成内容（Generator），“矩阵读者Agent”进行垂直纬度判别（Discriminator），“主编Agent”充当损失函数（Loss Function）进行仲裁，最终由人类介入进行全局梯度干预。

### 核心技术栈规范：

- 
- **状态调度**：纯Python控制流程，建议使用 transitions 库管理状态。
- **数据校验**：Pydantic 强制约定LLM输出格式。
- **LLM调用**：轻量级封装，使用原生API或 OpenAI SDK（兼容多模型接口），结合 instructor 或原生 Tool Calling 实现结构化输出。
- **Prompt管理**：Jinja2 模板引擎，实现Prompt与代码彻底解耦。
- **并发处理**：asyncio，读者Agent的评审必须并发执行以节省时间（注意并发频率限制，各个LLM供应商的API调用存在频率限制）。

## 3. 核心数据契约 (Data Contracts - Pydantic Models)

系统内Agent的通信必须通过严格的JSON格式。以下是核心数据结构的Pydantic定义规范，开发时需严格遵守。

### 3.1 读者反馈模型 (Reader Feedback)

每个读者Agent必须返回此结构。

codePython



```
from pydantic import BaseModel, Field
from typing import List, Optional

class Issue(BaseModel):
    error_type: str = Field(..., description="错误类型，如：OOC, Lore_Conflict, Pacing_Slow, AI_Tone")
    severity: int = Field(..., ge=1, le=5, description="严重程度1-5")
    quote: str = Field(..., description="原文引用")
    description: str = Field(..., description="问题详细描述及原因")

class ReaderFeedback(BaseModel):
    reader_role: str = Field(..., description="读者身份标识")
    immersion_score: int = Field(..., ge=1, le=10, description="本章沉浸感评分")
    emotional_watermark: str = Field(..., description="当前情绪水位，如：压抑、爽快、疲劳")
    issues: List[Issue] = Field(default_factory=list, description="发现的问题列表")
```

### 3.2 主编仲裁模型 (Editor Arbitration)

主编接收所有 ReaderFeedback 后，输出此结构。

codePython



```
class EditorRevisionPlan(BaseModel):
    pass_status: bool = Field(..., description="是否通过本次审核。True进入下一步，False打回重写")
    rejected_feedbacks: List[str] = Field(..., description="被主编驳回的读者意见理由（如：不符合本书题材）")
    revision_instructions: List[str] = Field(..., description="给作者的具体修改指令，要求具体、可执行")
    scene_target: str = Field(..., description="重写该场景的核心聚焦目标")
```

## 4. Agent 职能与配置规范 (Agent Specifications)

### 4.1 作者 (The Author)

- 
- **输入上下文**：book_meta + volume_outline + recent_summaries (滑动窗口) + ch_outline + editor_plan (若是重写)。
- **约束条件**：单次生成字数严格限制在 600-1000字（以Scene为单位）。强制遵循“Show, Don't Tell”原则。
- **大模型建议**：擅长网文语感的模型（如 DeepSeek-V3 / Kimi / 智谱）。

### 4.2 考据党读者 (The Lore Keeper)

- 
- **输入上下文**：characters.json + world_lore.json + Draft。
- **职责**：绝对理性。不关注文笔，只对比JSON设定树。寻找名字写错、死人复活、战力跨级等硬性逻辑BUG。

### 4.3 体验派读者 (The Pacing Junkie)

- 
- **输入上下文**：前2章全文本 + 第3-10章详细摘要 + Draft + book_meta中的 tone 字段。
- **职责**：维护情绪体验。使用 emotional_watermark 追踪情绪。如果发现“连续3章吃瘪”、“节奏水”、“黄金三章无看点”，立刻报 severity: 5 的致命错误。

### 4.4 排雷兵 (The Anti-Trope Scanner)

- 
- **输入上下文**：book_meta中的 forbidden_elements + Draft。
- **职责**：机械式扫描+体验式预警。只要出现违禁词汇或套路（如“嘴角勾起一抹冷笑”、“强行降智”），立即提取 quote 并报错。

### 4.5 主编 (The Editor)

- 
- **输入上下文**：book_meta + 所有读者反馈 + ch_outline + Draft。
- **职责**：宪法护卫队。过滤与 book_meta 冲突的读者意见，统合剩余意见，生成具体的 EditorRevisionPlan。
- **大模型建议**：必须使用逻辑推理最强、指令遵循最严的模型（如 Claude 3.5 Sonnet / GPT-4o）。

## 5. 工作流与状态机设计 (Workflow & State Machine)

核心工作流必须封装在一个具备熔断机制的状态机类中。

- 
- **状态枚举 (States)**:INIT: 加载元数据、角色卡、细纲。DRAFTING: 触发作者Agent生成正文（单次写一个Scene）。REVIEWING: **并发（Async）**触发三个读者Agent进行评审。EDITING: 触发主编Agent整合审查。HUMAN_INTERVENTION: 触发挂起，等待控制台人工输入指令（通过/修改细纲/强制重写）。COMMITTING: 将通过的草稿合并入正文库，触发 Summarizer 更新最近摘要和角色JSON状态。
- **流转逻辑与熔断 (Transitions & Circuit Breaker)**:DRAFTING -> REVIEWING -> EDITING。如果在 EDITING 中 pass_status == True，且场景未写完，回到 DRAFTING；若写完，进入 HUMAN_INTERVENTION 进行最终审查。如果在 EDITING 中 pass_status == False，重试计数器 +1，回到 DRAFTING。**【熔断机制】**：如果当前Scene的重试计数器 > 3，强行切断循环，流转至 HUMAN_INTERVENTION 抛出死锁警报。

## 6. 开发原则与编码规范 (Coding Guidelines)

### 6.1 LLM 客户端抽象层 (LLM Agnostic Pattern)

严禁在业务代码中直接写死 openai.ChatCompletion。必须实现一个基类：

codePython



```
class BaseLLMClient(ABC):
    @abstractmethod
    async def generate_text(self, system_prompt: str, user_prompt: str, **kwargs) -> str:
        pass
        
    @abstractmethod
    async def generate_json(self, system_prompt: str, user_prompt: str, response_model: Type[BaseModel]) -> BaseModel:
        # 此处必须实现基于 Pydantic 的强制 JSON 解析逻辑
        pass
```

### 6.2 Prompt 资产化 (Prompt as Code)

所有 Prompt 必须作为 .j2 文件独立存放，业务代码仅传递上下文参数：

codePython



```
# 示例：渲染主编的Prompt
template = jinja_env.get_template('editor_review.j2')
prompt = template.render(
    book_tags=book_meta.sub_genres,
    outline=current_outline,
    feedbacks=reader_feedbacks_json
)
```

### 6.3 健壮性与重试机制 (Robustness)

基于大模型输出的不稳定性，网络请求和JSON解析层必须使用 Tenacity 库包裹重试逻辑：

codePython



```
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
async def safe_json_generation(...):
    # 如果JSON解析失败（比如模型输出了markdown代码块外带废话），将触发重试
```

### 6.4 增量状态更新 (State Mutation Strategy)

只有在 COMMITTING 阶段，才能更新 characters.json 和 recent_chapters 目录。必须由一个专门的 StateUpdater Agent 负责提取最新一章的信息并进行 JSON 覆盖。更新前必须进行原文件备份（如存入 .backup/）。

## 7. 下一步开发建议 (Next Step Actionable)

阅读完此文档后，你的开发顺序应当是：

1. 
2. **基础设施**：搭建项目目录，安装依赖 (pydantic, transitions, jinja2, tenacity, asyncio 相关)。
3. **定义模型**：将本文档第3节的Pydantic模型在 src/core/models.py 中写出。
4. **实现LLM基类**：完成带重试机制的 generate_json 接口。
5. **Mock测试**：暂时不写LLM逻辑，用死数据（Mock）走通一次状态机流转流程，验证文件系统读写和版本号累加逻辑。
6. **Agent实装**：逐步编写Jinja2模板，挂载真实LLM API进行单场景测试。
